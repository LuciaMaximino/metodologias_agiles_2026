# Ejercicio Nro: 13

## Enunciado
1. ¿Qué es Extreme Programming (XP) y cuál es su objetivo principal dentro de las metodologías ágiles?
2. ¿Cuáles son los cinco valores principales de XP? Explicá brevemente cada uno.
3. ¿Por qué XP considera que las pruebas son un elemento fundamental del desarrollo de software?
4. ¿Qué es Test Driven Development (TDD) y cómo se relaciona con XP?
5. ¿En qué consiste la práctica de Pair Programming? Mencioná dos ventajas y una posible dificultad.
6. ¿Qué son las historias de usuario en XP y por qué se prefieren frente a una especificación extensa de requisitos?
7. ¿Qué significa Continuous Integration en XP y qué beneficios aporta al equipo de desarrollo?
8. ¿Cómo se aplica el concepto de Weekly Cycle en un proyecto desarrollado con XP?
9. En XP se plantea que se fija el tiempo, el costo y la calidad, y se negocia el alcance. ¿Qué significa esta idea? Explicalo con un ejemplo.
10. Elegí tres prácticas de XP y explicá cómo podrían aplicarse en un proyecto real de desarrollo de software.
## Resolución
1. Extreme Programming es una metodología ágil para desarrollar software que se adapta fácilmente a los cambios. Fue creada por Kent Beck y busca que los equipos trabajen de forma más eficiente y organizada. Su objetivo principal es reducir los riesgos del proyecto, responder rápidamente a los cambios, mejorar la productividad y lograr que el equipo trabaje de manera más cómoda y efectiva.

2. Los cinco valores de XP son:
-Comunicación: fomentar el diálogo constante entre los miembros del equipo y con el cliente.
-Simplicidad: hacer las soluciones lo más simples posible, evitando complicaciones innecesarias.
-Retroalimentación: obtener respuestas rápidas sobre el trabajo realizado para detectar errores y mejorar.
-Valentía: animarse a realizar cambios cuando son necesarios, aunque sean difíciles.
-Respeto: valorar el trabajo y las opiniones de todos los integrantes del equipo.

3. XP considera que las pruebas son esenciales porque ayudan a comprobar que el software funciona correctamente desde el principio. Las pruebas se crean junto con el código y permiten detectar errores más rápido. Además, le dan confianza al equipo porque demuestran que las funcionalidades desarrolladas funcionan como se espera.

4. TDD, desarrollo guiado por pruebas, es una técnica en la que primero se escribe una prueba y luego el código necesario para que esa prueba funcione. Es una práctica muy importante dentro de XP. El proceso consiste en:
-Escribir una prueba que falle.
-Programar lo necesario para que funcione.
-Mejorar el código sin cambiar su comportamiento.
Esto ayuda a desarrollar software de mejor calidad y más fácil de mantener.

5. Pair Programming consiste en que dos programadores trabajen juntos en una misma computadora para desarrollar el código. Lo ideal es que uno sea un programador novato o junior y que el otro sea un programador senior o con experiencia. Las parejas se tienen que ir rotando.
Ventajas:
-Permite compartir conocimientos y resolver problemas más rápido.
-Ayuda a detectar errores y mantener una mejor calidad del código.
Posible dificultad:
-Puede resultar incómodo si las personas no trabajan bien juntas o si no cuentan con un espacio adecuado para compartir la computadora.

6. Las historias de usuario son descripciones cortas de funcionalidades que el sistema debe tener. Generalmente se escriben en tarjetas, contienen una breve explicación de lo que necesita el usuario y un tiempo estimado sobre cuanto tiempo llevaria hacerla. Se prefieren a los documentos extensos de requisitos porque son más fáciles de entender, modificar y priorizar cuando cambian las necesidades del proyecto.

7. La Integración Continua consiste en integrar los cambios realizados por los programadores varias veces al día en una misma base de código. Sus beneficios son:
-Detectar errores rápidamente.
-Evitar problemas al unir el trabajo de distintos desarrolladores.
-Mantener el sistema funcionando correctamente en todo momento.

8. El Weekly Cycle divide el trabajo en ciclos de una semana. Al comienzo de la semana, el equipo planifica qué historias de usuario desarrollará. Luego se reparten las tareas, programan las funcionalidades y realizan las pruebas necesarias. Al finalizar la semana, las nuevas funcionalidades deberían estar listas para mostrarse o probarse.

9. Significa que no se deben aumentar los plazos, gastar más dinero ni reducir la calidad del software para terminar el proyecto. Si surge un problema, lo que se ajusta es la cantidad de funcionalidades que se entregarán. Por ejemplo si un sistema debe tener tres módulos y el tiempo no alcanza, se pueden entregar primero dos módulos bien hechos y dejar el tercero para una siguiente etapa, en lugar de retrasar el proyecto o entregar software de baja calidad.

10. Informative Workspace: colocar en una pizarra las tareas, historias de usuario y el avance del proyecto para que todos puedan ver el estado del trabajo.
Ten-Minute Build: configurar herramientas que compilen el sistema y ejecuten las pruebas automáticamente en menos de diez minutos cada vez que se agrega código nuevo.
Incremental Design: mejorar el diseño del software de manera gradual, realizando pequeños cambios continuos para mantener el código limpio y fácil de modificar.
